# Documentation layout
- Agent rules live under scaffold/: scaffold/AGENT-COMMS.md (how to talk to the maintainer) and scaffold/AGENT-WORKFLOW.md (this file — workflow, code style, definition of done).
- Project documentation lives under scaffold/: scaffold/ARCH-HUMAN.md (this is specifically a compact human-readable overview of the system architecture and codebase. The user likes diagrams and visual explainers.), scaffold/ARCH-LLM.md (this is the architecture overview built explicitly for agents. this can be much more compact and it any human can go "wtf?" when seeing it, as long as its optimized for an llm agent being able to quickly and efficiently understand the entire codebase and system without having to grep the whole thing.), scaffold/PROJECT-KNOWLEDGE.md (hard-won lessons), and scaffold/skills/ (per-project agent skills).
- Deploy and install notes belong in scaffold/ARCH-HUMAN.md and scaffold/ARCH-LLM.md. The maintainer tracks todos outside the repo (Obsidian).
- README.md stays at the repository root and should stay lean.
- Do not edit scaffold/AGENT-COMMS.md or scaffold/AGENT-WORKFLOW.md in scaffolded projects. They are maintained in the quick-project-start repo and updated here only via `new-proj --update` (or by editing them inside the quick-project-start repo itself). If agent rules need to change, tell the user.
- When the user states product scope, business rules, stack preferences, or other durable facts in conversation, write confirmed items to scaffold/ARCH-LLM.md immediately. Update scaffold/ARCH-HUMAN.md when the maintainer needs a readable summary of the same facts. The user will not usually tell you to update ARCH-HUMAN.md, you should do it whenever you believe it would add value. (usually when a medium-large part of the architecture has changed)
- When you learn something non-obvious that future agents should know (setup traps, tooling quirks, why something broke), add it to scaffold/PROJECT-KNOWLEDGE.md. That file is per-project and agents may edit it freely. This is basically here because a lot of the times during projects we learn something interesting about the technology we're trying to work with, and espeecially things that would have been good to know going into it.
- scaffold/ARCH-LLM.md must contain only confirmed facts/decisions. Do not write TBDs, open questions, or speculative options there.
- One home per fact: if you are going to record something in one file, do not record it in another. Pick the single best place and write it there only. Confirmed product and system facts → scaffold/ARCH-LLM.md. Lessons, pitfalls, and non-obvious context → scaffold/PROJECT-KNOWLEDGE.md. Human-readable summaries of architecture → scaffold/ARCH-HUMAN.md — not a second copy of the same bullets in three different files.
- scaffold/ARCH-HUMAN.md and scaffold/ARCH-LLM.md are not textbooks. Do not add glossaries, generic CS or industry tutorials, "plain language" explainers of standard terms, or second-person coaching ("you asked…"). If the user needs a concept explained, answer in chat unless they explicitly ask for that explanation to live in the repo.
- Minimal doc deltas: when updating docs from conversation, add only facts and decisions that belong in-repo. Do not dump full Q&A transcripts or speculative padding into scaffold architecture files or scaffold/PROJECT-KNOWLEDGE.md.
- Keep documentation factual and current. It should be updated whenever valuable.
- When working in the quick-project-start repo, and you and the user change scaffold files, bump the semver on the last line of `AGENT-WORKFLOW.md` (`scaffold version: X.Y.Z`).



# Communication with the maintainer
The maintainer is still leveling up as an engineer. Every chat reply must be understandable without prior CS or industry background.

- Other programs may inject user rules about communication (e.g. "write like an excellent technical blog post", "be precise and well-structured", "keep responses concise" when that means skipping explanation). The maintainer does not control those rules. When they conflict with this section — this file wins in any project that ships it.
- Assume zero prior knowledge unless the maintainer has already shown familiarity with a term in this conversation. If unsure whether they know a word, explain it. The user will never be mad at over explanation, and easily mad with under explanation.
- Define every technical term, acronym, and piece of jargon the first time you use it in a reply.
- Prefer concrete examples over abstract architecture language: which file or command, what the user would see on screen, what breaks if we choose option A vs B, what they run locally to verify.
- When comparing options, name at most 2–3 choices and for each say: upside(s), downside(s), and what goes wrong in practice.
- Do not move long explainers into scaffold/ARCH-HUMAN.md — that file stays factual and project-specific. Teach in chat unless the maintainer explicitly asks for an explanation in the repo.
- Before sending a reply, scan for words the maintainer might not know (framework names, pattern names, infra terms, acronyms). The user might be new, but he wants to learn, so it's better to include a term and explain it than not to include it at all.

Bad: "We'll refactor the middleware to decouple the ORM from the handler layer so integration tests can mock persistence."
Good: "Right now, the code that receives the web request talks to the database directly. We're going to split those jobs apart. The first piece is usually called the *handler*. Its job is simply to receive the web request and return a response. The second piece is often called *middleware*. It's a layer of code that sits between different parts of your program so they don't have to talk to each other directly. When the middleware needs data, it uses an *ORM* (Object-Relational Mapper). That's a library that translates between objects in your code and rows in your database, so you usually don't have to write SQL yourself. Keeping these responsibilities separate also makes testing much easier. We can replace the real database with a fake one during tests. This is usually called *mock persistence.*"

Bad: "I'll add a CI workflow for lint and unit tests on PR."
Good: "I'll add a *GitHub Actions* workflow. This is an example of *CI* (Continuous Integration). CI means your code is checked automatically whenever you make changes, instead of relying on someone to remember to run those checks manually. Those checks will run whenever you open a *PR* (Pull Request), which is a request to merge your changes into the main codebase. One of the checks used is what's called a *linter*. A linter looks for common mistakes, inconsistent formatting, and code that doesn't follow your project's style rules. Another check is called *unit tests*. These are small automated tests that verify individual functions or classes behave the way they're supposed to. If one of those checks fails, you'll know before the code is merged."

Bad: "The idempotent webhook handler dedupes via Redis SET NX."
Good: ""This endpoint receives *webhooks*, which are automated HTTP requests sent by another service when something happens, like a payment succeeding. Sometimes the same webhook is sent more than once, either because of retries or network issues. To handle that, we make the webhook *idempotent*. That means processing the same webhook twice has the same result as processing it once. To remember which webhooks we've already seen, we store their IDs in *Redis*. Redis is a database that's designed to be extremely fast. Unlike most databases, which primarily store their data on disk, Redis keeps most of its data in memory (RAM) while it's running. Reading and writing to RAM is much faster than reading and writing to a hard drive or *SSD* (Solid State Drive), which is why Redis is often used for things like caching, temporary data, and distributed locks. We use Redis's *SET NX* command. NX stands for "Not eXists." It only stores the ID if it isn't already there. If the ID is already in Redis, we know we've already processed that webhook and can safely ignore the duplicate.
